PRODUCT REQUIREMENTS DOCUMENT Simulacrum Feedback &amp; Social Proof System User Feedback Collection · Admin Moderation · Home Page Testimonial Display Document ID SIM-PRD-FBK-001 Version 1.0 — Draft for Engineering Feature Group Feedback Collection · Admin Moderation · Social Proof Display Parent PRD Simulacrum PRD v1.0 · SIM-PRD-PROF-001 Platform Flask + SQLAlchemy + MySQL · Python · GoDaddy Author Jean Moussa Delphonse · PrezentEnergy Ventures Date April 2026 This PRD specifies three interrelated features: (1) a user-triggered feedback form embedded in the authenticated dashboard capturing star rating, outcome report, layer attribution, and testimonial quote; (2) an admin moderation queue where submitted feedback is reviewed and approved or rejected before going live; and (3) a social proof section on the Simulacrum home page displaying approved testimonials in a format designed to convert both individual users and potential referral partners. Bayesian social proof: over time as feedback volume grows, the home page display will surface the most relevant testimonials to each visitor segment — partner referrals see partner-client success stories first; direct visitors from specific industries see domain-matched testimonials first. This is the Explore/Exploit transition applied to conversion optimization: early phase shows all approved testimonials (explore); later phase concentrates on the highest-converting subset for each segment (exploit). 1. User Feedback Collection 1.1 Overview Feedback submission is entirely user-triggered. Simulacrum does not send automated prompts, pop-ups, or email nudges asking for feedback. Instead, a persistent &apos;Share Your Story&apos; entry point is available in the authenticated dashboard at all times. This respects the user&apos;s agency and ensures that feedback submitted is intentional and genuine rather than prompted by timing logic. The feedback form captures three components: a star rating (1–5), an outcome report (what specifically changed or was achieved, attributed to a layer), and a testimonial quote (a short statement the user is comfortable having shown publicly). These three components together produce the richest possible social proof — a credible claim with a specific result and a human voice behind it. The most persuasive testimonials answer three questions in sequence: who you were before, what you did, and what changed. The feedback form is designed around this structure. The outcome field captures &apos;what changed.&apos; The layer attribution captures &apos;what you did.&apos; The testimonial quote captures the user&apos;s own voice on the transformation. Together they create a narrative arc that converts skeptical visitors more effectively than a star rating alone. 1.2 Feedback Entry Points Entry Point Specification Dashboard sidebar &apos;Share Your Story&apos; link in the authenticated dashboard sidebar navigation, below &apos;My Simulations&apos; and above Settings. Always visible. Opens the feedback modal. Simulation result page A subtle &apos;Share your results&apos; link in the footer of any Simulation view page. Opens the same feedback modal, with the current Simulation pre-selected. Layer action result card A &apos;This worked — share it&apos; micro-link below any completed agent action result card. Opens the feedback modal with the relevant layer pre-selected. Profile settings A &apos;Testimonials&apos; sub-section in Profile &amp; Bio settings showing the user&apos;s prior feedback submissions and their moderation status. A &apos;Submit new feedback&apos; button opens the modal. 1.3 Feedback Form — Field Specification The feedback form renders as a modal popup consistent with the Interactive Orchestrator Diagram modal — white background, centered, closeable via × button, Escape key, or overlay click. No automated prompts, pop-ups, scheduled emails, or nudges are sent to encourage submission. The &apos;Share Your Story&apos; entry point is always visible in the dashboard sidebar for authenticated users. FR-FB-02: The feedback form modal is consistent with the Simulacrum modal pattern: white background, centered, closeable via × button, Escape key, or overlay click. FR-FB-03: Star rating, outcome text, testimonial quote, and name display preference are all required fields. The form cannot be submitted with any of these blank. Layer attribution is optional. FR-FB-04: On successful submission, the modal shows a confirmation screen and the user receives a SendGrid confirmation email. The admin receives a notification email. The edit updates the existing record (not a new submission). The admin sees the latest version. The submitted_at timestamp reflects the original submission; an updated_at timestamp reflects the last edit. FR-FB-07: The Expertise Zone of the user&apos;s Simulation (if a Simulation is associated with the feedback) is stored on the feedback record at submission time as a snapshot — not a live FK reference — so that if the Simulation is later deleted, the feedback record retains its zone context for display purposes. 2. Admin Moderation Queue 2.1 Overview Every submitted feedback record enters a Pending state and appears in the Admin Moderation Queue before any public display. The admin reviews each submission, reads the outcome text and testimonial quote, checks the star rating and layer attribution, and makes one of three decisions: Approve (publish to home page), Reject (notify user with optional reason), or Feature (approve and pin to top of home page display). The moderation queue is accessible from the Admin Panel at /admin/feedback. It is separate from the revenue dashboard and partner management sections but follows the same admin panel navigation pattern. No feedback ever appears publicly without an explicit admin Approve or Feature action. 2.2 Moderation Queue — Layout UI Element Specification Queue header Title: &apos;Feedback Moderation&apos;. Three tab filters: Pending (count badge), Approved, Rejected. Default view: Pending. A &apos;Search&apos; input filters by user name, quote text, or layer. Feedback card Each submission renders as a card showing: star rating (gold stars), submitted by (display name + username), submission date, layer(s) attributed (colored badge per layer), outcome text (full), testimonial quote (full, in quotation marks), name display preference, and Simulation name if linked. Action buttons Three buttons per card: &apos;Approve&apos; (teal), &apos;Feature&apos; (gold — approve and pin to top), &apos;Reject&apos; (red). Clicking Approve or Feature immediately moves the record to Approved status. Clicking Reject opens a rejection modal. Rejection modal A small modal with an optional &apos;Reason for rejection&apos; textarea (max 300 chars) and a Confirm Reject button. The reason is stored in admin_note and included in the rejection email to the user. Reason is optional — admin can reject without providing one. Bulk actions Checkboxes on each card enable bulk Approve or Bulk Reject from the Pending tab. Useful when multiple submissions arrive from a launch event. Approved tab Lists all approved feedback with approve date, approver name, is_featured flag, and display_order. Admin can toggle Featured status and adjust display order (drag-drop or numeric input) from this tab. An &apos;Unpublish&apos; button returns a record to Pending status without deleting it. Stats bar Above the queue: total submitted (all time), pending count, approved count, rejected count, average star rating (approved only), most attributed layer. 2.3 Moderation Actions &amp; Notifications Action Record Update User Notification (SendGrid) Approve status → approved. approved_by, approved_at populated. is_featured = False. No email sent to user on unpublish. The testimonial simply no longer appears on the home page. The user&apos;s Testimonials panel still shows the record as Approved. 2.4 Admin API Endpoints — Moderation Method Endpoint Description GET /api/admin/feedback List all feedback with filter (pending/approved/rejected) and search. GET /api/admin/feedback/stats Aggregate stats: total, pending, approved, rejected, avg rating, top layer. PUT /api/admin/feedback/:id/approve Approve a feedback record. Sets status, approved_by, approved_at. PUT /api/admin/feedback/:id/feature Feature a feedback record. Sets is_featured=True plus approval fields. PUT /api/admin/feedback/:id/display-order Set display_order integer for approved record. Used by drag-drop reorder. PUT /api/admin/feedback/reorder Bulk update display_order for all approved records after drag-drop. POST /api/admin/feedback/bulk-approve Approve multiple records by ID array. POST /api/admin/feedback/bulk-reject Reject multiple records by ID array with optional shared admin_note. 2.5 Functional Requirements — Admin Moderation FR-MOD-01: No feedback record appears publicly without an explicit admin Approve or Feature action. Pending and Rejected records are never visible on the home page or any public-facing surface. FR-MOD-02: The admin moderation queue is accessible at /admin/feedback and is protected by admin-role authentication. Regular user sessions cannot access this route. FR-MOD-03: Approve, Feature, and Reject actions take effect immediately. Approved and Featured records appear on the home page within one page render cycle (no cache delay beyond standard page caching). FR-MOD-04: Rejection emails to users always include an invitation to submit new feedback. The platform never communicates that a user is banned from submitting feedback, even if the admin has privately flagged an account. FR-MOD-05: The admin can reorder approved testimonials via drag-drop in the Approved tab. The display_order field determines home page card order. Featured records always render above non-featured records regardless of display_order. FR-MOD-06: The admin can unpublish any approved record at any time without deleting it. The record returns to Pending status and can be re-approved later. Unpublishing does not notify the user. FR-MOD-07: The admin notification email for new submissions includes the full testimonial quote, outcome text, star rating, and layer attribution so the admin can evaluate the submission without opening the admin panel. 3. Home Page Social Proof Display 3.1 Overview The social proof section appears on the Simulacrum home page (simulacrum.io) below the hero section and above the feature breakdown. It serves two audiences simultaneously: individual professionals evaluating whether Simulacrum can work for them, and potential referral partners (coaches, advisors) evaluating whether to send their clients to the platform. The design must speak to both without alienating either. The section displays approved testimonials as cards. Featured testimonials appear first. The display rotates through all approved testimonials using a horizontal scroll carousel on desktop and a stacked card view on mobile. The section also includes two aggregate trust signals: an overall star rating average and a total count of Simulations generated to date. Explore/Exploit applied to social proof display: in the early platform phase (fewer than 20 approved testimonials), all approved records are displayed in admin-set order. Once the pool exceeds 20, the platform enters an optimization phase: a visitor arriving from a referral partner link sees testimonials from users referred by partners first; a visitor from a LinkedIn post sees testimonials attributed to the layer most relevant to the post&apos;s topic. This segment-matched display is implemented via a UTM parameter read on page load and a server-side testimonial ranking call. 3.2 Section Layout — Desktop Element Specification Section header Two-line header: (1) large bold: &apos;Real results from real professionals.&apos; (2) subtitle in muted text: &apos;Every story below came from someone who uploaded a resume and ran their first Simulation.&apos; This framing sets the expectation that these are authentic user outcomes, not marketing copy. Trust signals bar A horizontal bar below the header showing two metrics side by side: average star rating (e.g. &apos;4.9 ★ average rating&apos;) and simulation count (&apos;2,847 Simulations generated&apos;). Both are pulled live from the database. Star count uses approved feedback only. Simulation count uses the total simulations table count. Featured testimonial If one or more records have is_featured=True, the first featured record renders as a large hero card spanning the full section width. Dark navy background, large quote text in white, gold quotation marks, user&apos;s display name and Expertise Zone shown below. The layer badge(s) are shown as small colored chips. Carousel cards Non-featured approved testimonials render as a horizontal scrolling carousel of cards. Each card: white background with subtle shadow, gold star rating, testimonial quote in dark text (large enough to read without glasses), display name, Expertise Zone name (from Simulation snapshot), layer badge(s), and submission date (month and year only). Card count visible 3 cards visible at once on desktop (1200px+). 2 cards on tablet (768–1199px). Chevron arrows for left/right navigation. Dots indicator below showing position in carousel. Auto-rotation Carousel auto-advances every 6 seconds when idle. Pauses on hover or keyboard focus. Resumes 3 seconds after hover ends. Partner CTA Below the carousel, a subtle two-column row: left side for users (&apos;Ready to write your story? Create your first Simulation →&apos;), right side for partners (&apos;Are you a coach or advisor? Join our partner program →&apos;). This is the only place on the home page where both audiences are explicitly addressed side by side. 3.3 Testimonial Card — Field Mapping Card Element Source Field Display Rules Star rating user_feedback.star_rating Gold filled stars. Always shown. 1–5 stars rendered as SVG icons. Quote text user_feedback.quote_text Shown in quotation marks. On each page load, a UTM parameter check and optional referrer analysis determines which testimonial subset to prioritize in the carousel. The full pool is still accessible by scrolling — only the initial visible cards are ranked. Visitor Signal Testimonial Ranking Priority utm_source=partner&amp;utm_medium=referral Partner-referred testimonials first (is_partner_referred=True). Featured records always first regardless. utm_source=linkedin Testimonials with L1 or L2 layer attribution prioritized (consulting and coaching audiences tend to arrive via LinkedIn). utm_source=twitter or utm_source=x Highest star rating testimonials first (social media visitors are more skeptical — lead with strongest social proof). Direct / no UTM Admin display_order used. Featured first, then display_order ascending. Repeat visitor (cookie present) Rotate to show testimonials not yet seen by this visitor. Tracks shown testimonial IDs in a session cookie. Resets after 30 days. 3.5 Home Page API Endpoints Method Endpoint Description GET /api/feedback/public Return approved testimonials for home page. Accepts utm_source query param for segment ranking. Returns is_featured records first, then display_order. Max 50 records. GET /api/feedback/stats/public Return aggregate trust signals: avg_rating (approved only), total_simulations (all time), approved_count. POST /api/feedback Submit new user feedback. Auth required. Validates all required fields. Creates UserFeedback record with status=pending. Dispatches confirmation emails async. GET /api/feedback/mine Return all feedback submissions by the authenticated user with their moderation status. PUT /api/feedback/:id Edit a pending feedback submission (user only, own records only, status must be pending). 3.6 Functional Requirements — Home Page Display FR-HP-01: The social proof section renders on the home page server-side via Jinja2. Testimonial data is fetched from /api/feedback/public at page render time and injected into the template context. The section does not require client-side JavaScript to render initial content — it is visible without JS enabled. FR-HP-02: Featured testimonials (is_featured=True) always render first, above non-featured testimonials, regardless of utm_source or display_order. If multiple records are featured, they render in admin-set display_order among themselves. FR-HP-03: The carousel auto-advances every 6 seconds when idle. Auto-advance pauses on hover or keyboard focus and resumes 3 seconds after interaction ends. Users can navigate manually using chevron buttons or dot indicators. Keyboard navigation is supported (left/right arrow keys when carousel is focused). FR-HP-04: The aggregate trust signals (average rating and Simulation count) are fetched live from the database on each home page render. They are not cached for more than 60 seconds to ensure they reflect recent activity. FR-HP-05: Testimonial cards never display the user&apos;s email address, account username, or any financial data from their Simulation. The only personal information shown is the display name as set by the user&apos;s name_display preference. FR-HP-06: The &apos;Via [partner name]&apos; attribution on testimonial cards is shown only when: (a) the user was referred by an approved partner, (b) the partner has a published public profile, and (c) the partner has not opted out of testimonial attribution in their partner settings. FR-HP-07: The segment-matched display (Phase 2) is implemented only when the approved testimonial pool exceeds 20 records. Below this threshold, the platform uses standard admin display_order. The threshold is configurable in platform_settings under &apos;feedback_segment_threshold&apos;. FR-HP-08: The home page section is fully responsive. On mobile (under 768px), the carousel becomes a stacked vertical list of cards. Auto-rotation is disabled on mobile to avoid accidental swipes. The trust signals bar stacks vertically (rating above simulation count). 4. User-Facing Feedback Management 4.1 Testimonials Panel in Profile Settings The Testimonials sub-section in Profile &amp; Bio settings gives authenticated users a complete view of all feedback they have submitted and the current moderation status of each. This is the only place users can track the lifecycle of their submissions. Element Specification Submission list All of the user&apos;s feedback records listed in reverse chronological order (most recent first). Each row shows: star rating, quote excerpt (first 80 chars), submission date, and status badge (Pending = amber, Approved = teal, Featured = gold, Rejected = red). Status badge detail Hovering the Approved or Featured badge shows the approval date. Hovering the Rejected badge shows the admin_note if one was provided, or &apos;No reason provided&apos; if admin_note is null. Edit (Pending only) A pencil icon on Pending records opens the feedback modal pre-filled with the current values. The user can edit quote_text, outcome_text, layers_attributed, and name_display. Rating cannot be changed after submission. Save triggers a PUT to /api/feedback/:id. Live badge Approved and Featured records show a &apos;Live on home page&apos; chip with a direct link to simulacrum.io (the home page social proof section). Submit new button &apos;Submit new feedback&apos; button at the top of the Testimonials panel. Opens a blank feedback modal. Always available regardless of prior submission status. Withdraw A &apos;Withdraw&apos; option on Approved and Featured records allows the user to request removal of their testimonial from public display. This creates an admin notification and sets a withdrawn_requested_at field. The record remains Approved until the admin acts — the admin can honor the withdrawal (Unpublish) or contact the user. 4.2 Functional Requirements — User Feedback Management FR-UFM-01: The Testimonials panel in Profile Settings shows all feedback records submitted by the authenticated user. Records are never deleted from this view — only their status changes. FR-UFM-02: Pending records can be edited by the user. The edit updates the existing record in place. Star rating cannot be changed after initial submission. All other fields can be edited while the record is in Pending status. FR-UFM-03: Approved and Featured records cannot be edited by the user. If the user wants to change their testimonial after approval, they must submit new feedback and request withdrawal of the approved record. FR-UFM-04: The withdrawal request sets withdrawn_requested_at on the record and sends an admin notification email. The record remains live until the admin takes action. The user&apos;s Testimonials panel shows a &apos;Withdrawal requested&apos; status on the record. 5.
