1. Executive Summary
Simulacrum is a web application that allows authenticated users to upload their professional resume and have an AI analyze their career history to identify zones of excellence, extractable intellectual property, and monetizable expertise. The platform then generates one or more Simulations — structured, AI-powered blueprints that map a user's real work deliverables onto a five-layer wealth automation framework, producing a personalized roadmap for turning career capital into continuous residual income.
Each Simulation is a living object that users can iterate on, evolving from a simple 1:1 active income model all the way to a fully automated Layer 5 wealth engine with SaaS products, licensing deals, and passive investment vehicles — all anchored to verifiable professional history captured in the resume or LinkedIn profile.

2. Problem Statement & Vision
The Problem
Most professionals accumulate 10–30 years of experience that never compounds beyond their next salary negotiation. Their intellectual property — battle-tested frameworks, domain knowledge, repeatable methodologies — is locked inside their employment history, invisible even to themselves. A life career coach can unlock this, but coaching is expensive, time-intensive, and does not scale.
The Vision
Simulacrum democratizes the career coaching process and adds a wealth-engineering layer on top. By combining resume parsing, LinkedIn profile crawling, AI-powered expertise extraction, and a structured five-layer income model, the platform helps any professional answer three questions:
What are you excellent at — and what real deliverables prove it?
How is that excellence packaged into sellable products?
How do those products eventually run without you?
Strategic Positioning
Simulacrum sits at the intersection of career technology, AI coaching, and wealth strategy. Unlike resume builders or job boards, it is not about finding the next role. It is about converting the totality of a professional's career into a compounding economic engine.

Term

Simulation
A complete wealth-building blueprint generated for a user from their resume or LinkedIn profile and a chosen Expertise Zone. One user may create multiple Simulations for different career verticals. Each Simulation is independently charged at the current admin-configured price (default $10.00).
Expertise Zone
A discrete area of professional strength extracted by the AI — e.g., 'Enterprise Data Pipeline Architecture' or 'B2B SaaS Go-to-Market.' Grounded in verifiable Deliverables.

A real, named output from the user's career history used as AI evidence for an Expertise Zone.

One of five income tiers in a Simulation, ranging from active 1:1 work (Layer 1) to fully automated residual systems (Layer 5).

A specific monetizable product or service within a Layer, e.g., 'Data Architecture Workshop' or 'AI Audit SaaS Tool.'

Claude's explicit explanation of why a specific Income Stream was recommended for this user, surfaced inline in the UI alongside each recommendation.
Collaboration
The ability for a user to share a Simulation with one or more collaborators who can view, comment on, or co-edit layers, seeded from a prior Simulation session.
LinkedIn Crawl
An automated scrape of the user's public LinkedIn profile to extract work history, roles, and achievements as an alternative to resume upload.
Automation Blueprint
The Layer 4 and Layer 5 plan detailing tools, funnels, platforms, and investment vehicles the user should build to achieve residual income.
Fintech Toggle
An admin-controlled on/off flag that enables or disables live fintech API integration (Plaid, Alpaca, etc.) for Layer 5 wealth deployment recommendations.

5. Functional Requirements
5.1 Authentication & User Management
FR-AUTH-01: Email and password registration with Flask-Login and Bcrypt hashing (cost factor 12+).
FR-AUTH-02: Email verification required before first use. Password reset via tokenized email link (24-hour expiry).
FR-AUTH-03: Optional Google OAuth via Flask-Dance. JWT tokens for session management with 7-day expiry.
FR-AUTH-04: User profile stores full name, email, created_at, simulation_count, and total_spend.
5.2 Monetization — Per-Usage Fee
FR-MON-01: Each Simulation generation is charged at the current configured Simulation price, set by the admin in the platform settings. The default price at launch is $10.00. The price is stored in the platform_settings table under the key 'simulation_price' and is read dynamically at checkout — no redeployment required to change it.
FR-MON-02: Admin can update the Simulation price at any time via the Admin Panel. The new price takes effect immediately for all subsequent Simulation creations. In-progress checkouts (Stripe session already opened) honor the price at session creation time.
FR-MON-03: The Admin Panel displays the current configured price prominently alongside a price history log showing every price change: old value, new value, changed by (admin user_id), and timestamp.
FR-MON-04: The price shown to the user on the Simulation setup screen, payment confirmation dialog, and invoice is always fetched live from platform_settings — never hardcoded in the frontend. If the price has been changed since the user last visited, a session refresh surfaces the updated price before checkout.
FR-MON-05: Stripe Checkout integration. User must have a saved payment method or complete checkout before generation begins. The Stripe price object is created dynamically per transaction using the current simulation_price value — no pre-configured Stripe Price ID is used, allowing price changes without Stripe dashboard intervention.
FR-MON-06: Payment confirmation triggers the Simulation generation job via Celery task queue.
FR-MON-07: Failed generations (Claude API error after payment) trigger automatic refund via Stripe API within 60 seconds. The refund amount matches the exact amount charged, regardless of any price change that may have occurred between charge and failure.
FR-MON-08: Invoice emailed automatically for each successful Simulation, showing the price paid at time of purchase. Admin dashboard shows revenue, refund rate, per-user spend, and a price change audit trail.
5.3 Resume Management
FR-RES-01: Accept PDF and DOCX files up to 5 MB. Parse using PyMuPDF (PDF) and python-docx (DOCX).
FR-RES-02: Store original file on GoDaddy file system. Store parsed text in MySQL. Display parsed text in editable rich-text area for user correction before AI analysis.
FR-RES-03: Support multiple resumes per user. Each resume labeled with a version name and upload date.
FR-RES-04: Claude expertise zone extraction cached in Resume.expertise_zones JSON column. Re-called only if parsed_text changes.
5.4 LinkedIn Profile Integration
FR-LI-01: User authenticates with LinkedIn OAuth 2.0. Platform crawls the public profile: work experience, role titles, dates, and listed accomplishments.
FR-LI-02: Crawled data is normalized into the same parsed_text schema used by resume uploads, enabling identical downstream AI processing.
FR-LI-03: User can review and edit crawled LinkedIn data before triggering expertise zone extraction — same editable rich-text flow as resume.
FR-LI-04: LinkedIn data stored as a Resume record with source='linkedin' flag. Treated identically to uploaded resumes for Simulation creation.
FR-LI-05: Re-sync option allows user to re-crawl their LinkedIn profile to capture profile updates.
5.5 Simulation Management
FR-SIM-01: User selects a resume or LinkedIn profile, optionally provides a focus hint, and chooses an Expertise Zone. Platform displays the current configured Simulation price and charges that amount before generation begins.
FR-SIM-02: Generation streams via Server-Sent Events (SSE). Each layer appears in the UI as Claude completes it — users see the wealth pyramid populate in real time.
FR-SIM-03: Each Income Stream recommendation is displayed alongside Claude's AI reasoning — a 2–3 sentence explanation of why this stream was chosen for this user's specific background.
FR-SIM-04: Users can refine any individual Layer with a new constraint. Regeneration of a single layer is free (no additional charge).
FR-SIM-05: Users can create multiple Simulations from the same or different source documents. Each Simulation is independently named and managed on the Dashboard.
FR-SIM-06: Simulations persist indefinitely. Users can return to view, refine, or share any prior Simulation.
5.6 Collaboration
FR-COL-01: Any Simulation can be shared with collaborators via a unique share link. Collaborators do not pay to view a shared Simulation.
FR-COL-02: Collaboration can be seeded from a prior Simulation: the owner selects an existing Simulation as a starting point and invites collaborators by email.
FR-COL-03: Collaborator permission levels: Viewer (read-only), Commenter (can annotate layers), Editor (can trigger layer refinement — owner pays).
FR-COL-04: Collaboration activity (comments, refinements) logged in an audit trail visible to the Simulation owner.
FR-COL-05: Owner can revoke access at any time. Shared links expire after 30 days unless renewed.
5.7 Fintech Integration Toggle (Layer 5)
FR-FIN-01: By default, Layer 5 (Wealth Deployment) provides general guidance: recommended asset classes, platforms, and investment strategies tailored to the user's income profile.
FR-FIN-02: Admin-controlled on/off toggle in the platform settings enables live fintech API integration when activated.
FR-FIN-03: When enabled, Layer 5 may surface real fund options, brokerage account links, or portfolio suggestions via Plaid, Alpaca, or equivalent API.
FR-FIN-04: When fintech toggle is off, Layer 5 renders static wealth strategy content only. No external API calls are made.
FR-FIN-05: Toggle state is stored in a platform_settings table, not hardcoded. Change takes effect immediately without redeployment.
6. The Simulation Wealth Engine — Five Layers (Agent-Enhanced)
Every Simulation maps the user's Expertise Zone across five progressive income layers. Each layer is an active agent workspace — not passive guidance. For every layer, the Simulacrum agent plans and executes specific actions on the user's behalf, prompts for any missing information before acting, and displays structured results inline with full artifact storage. The agent operates using the Claude API with tool-use capabilities, a growing set of platform integrations, and a persistent action log per Simulation.
Agent execution follows a three-step loop for every action: (1) the agent determines what information it needs and surfaces an inline prompt form if anything is missing, (2) the agent executes the action using available tools or generates a production-ready artifact, and (3) the result is displayed inline with a status badge (Complete / In Progress / Failed), a timestamp, and action-specific controls (Copy, Download, Send, Regenerate). All outputs are stored in the Simulation's artifact store linked to a CHAR(9) action_id.
Agent action execution is included in the Simulation fee charged at creation time. There is no per-action charge. Re-execution of individual actions is free and archives prior outputs — it does not overwrite them. A new charge at the current configured price applies only when creating a brand-new Simulation.

7. Data Models (SQLAlchemy / MySQL)
7.1 ID Generation Standard — All Tables
Every primary key across all database tables is a generated 9-character alphanumeric string (CHAR(9)), not an auto-incrementing integer. The ID is generated at the application layer before INSERT using a cryptographically secure random function drawing from uppercase letters, lowercase letters, and digits (62-character alphabet). This applies to all id columns and all foreign key references.
Property
Specification
Type
CHAR(9) — fixed-length, NOT NULL, PRIMARY KEY
Alphabet
A-Z, a-z, 0-9 — 62 characters total
Length
9 characters — 62^9 = ~13.5 trillion unique values
Generation
secrets.token_urlsafe() sliced and filtered at application layer (Python)
Collision check
SELECT COUNT(*) before INSERT. Regenerate on collision (astronomically rare).
Foreign keys
All FK columns are also CHAR(9), indexed. No integer sequences anywhere in schema.
Example value
A3x9Kq2Rp, mB7nPw4Tz, Zr1Ys6Jd8
# utils/id_gen.py — 9-character alphanumeric ID generator
import secrets, string
ALPHABET = string.ascii_letters + string.digits  # 62 chars: A-Z a-z 0-9
ID_LENGTH = 9
def generate_id() -> str:
"""Return a cryptographically random 9-character alphanumeric string."""
return ''.join(secrets.choice(ALPHABET) for _ in range(ID_LENGTH))
# Usage in SQLAlchemy model (all tables follow this pattern):
from utils.id_gen import generate_id
class User(db.Model):
__tablename__ = 'users'
id = db.Column(db.CHAR(9), primary_key=True, default=generate_id, nullable=False)
# ...
class Simulation(db.Model):
__tablename__ = 'simulations'
id        = db.Column(db.CHAR(9), primary_key=True, default=generate_id, nullable=False)
user_id   = db.Column(db.CHAR(9), db.ForeignKey('users.id'), nullable=False)
resume_id = db.Column(db.CHAR(9), db.ForeignKey('resumes.id'), nullable=False)
# ...
# MySQL DDL equivalent:
# id CHAR(9) NOT NULL PRIMARY KEY  -- no AUTO_INCREMENT
# user_id CHAR(9) NOT NULL,
# FOREIGN KEY (user_id) REFERENCES users(id)

9.3 SSE Streaming Architecture
The /api/simulations/:id/stream endpoint uses Flask's Response with stream_with_context and Server-Sent Events. Claude's API streaming mode is used (stream=True). As each layer's JSON object is completed, it is parsed and emitted as a distinct SSE event. The frontend listens with EventSource and appends each layer to the UI as it arrives.
SSE event types emitted during generation:
layer_start — layer number and name, signals UI to render skeleton card
layer_data — complete layer JSON including all income_streams with ai_reasoning
layer_error — error on a specific layer, triggers retry UI for that layer
simulation_complete — all five layers delivered, marks Simulation as complete
simulation_error — fatal error, triggers refund process via Stripe API

15. Referral Partner & Advisor Access Program
Simulacrum supports a structured Referral Partner Program enabling life coaches, financial advisors, and career coaches to refer their clients to the platform and earn a commission on every Simulation their referred clients generate. Separately, clients may grant their advisor full read-only access to their Simulation for the purpose of coaching sessions. These two features — referral commission tracking and advisor view access — are distinct but architecturally linked through the ReferralPartner entity.
15.1 Program Overview
Property
Specification
Commission Model
Percentage of each Simulation charge earned by the referring partner. Applied at the moment of successful payment. Default global rate: 20%. Admin-configurable in platform_settings under the key 'partner_commission_rate'.
Rate Governance
Single global rate applies to all partners. Admin sets and adjusts the rate via the Admin Panel. Rate changes apply to all future Simulation charges — commissions already logged at a prior rate are not retroactively adjusted.
Payout Method
Stripe Connect. Partners must complete Stripe Connect onboarding before any commission is disbursed. Earnings accumulate in the partner ledger until the next payout cycle.
Payout Schedule
Admin-configurable in platform_settings under 'payout_schedule': weekly, bi-weekly, or monthly. Payout is triggered automatically by a Celery scheduled task on the configured cadence.
Referral Attribution
Each partner receives a unique 9-character alphanumeric referral code (CHAR 9, generated by the same id_gen utility). Attribution is captured via URL parameter (?ref=CODE) on signup. Stored as a session cookie persisting through registration and first checkout.
Attribution Window
30 days from first click. If a user clicks a referral link, registers within 30 days, and generates a Simulation within 30 days of registration, the referring partner earns the commission.
Eligible Events
Commission is earned on every Simulation generated by a referred client — not just the first. A referred client who generates 10 Simulations earns their referring partner 10 commission events.
Advisor View Access
Any registered partner may be granted full read-only access to a client's Simulation. The client initiates the share. The advisor sees all 5 layers, all income streams, all AI reasoning, and all agent action artifacts. No edit or regeneration capability.
Partner Types
Life Coach, Financial Advisor, Career Coach, Business Coach, Other. Type is self-declared at onboarding and displayed in the Admin Partner roster for filtering.
Partner Status
Pending (applied, awaiting admin approval), Active (approved, earning commissions), Suspended (commissions paused by admin), Inactive (self-deactivated).

15.3 Commission Calculation & Ledger
Commission is calculated and logged at the moment a referred client's Stripe payment is confirmed. The calculation uses the exact amount charged and the commission rate active at that moment — both values are stored on the commission record for immutability.
# Commission calculation (services/commission_service.py)
def calculate_commission(simulation_charge_amount: float) -> float:
rate = get_platform_setting('partner_commission_rate')  # e.g. 0.20
return round(simulation_charge_amount * rate, 2)
# Example: simulation_price = $12.00, rate = 20%
# commission = round(12.00 * 0.20, 2) = $2.40
# Commission record written on Stripe payment_intent.succeeded webhook:
# commissions table:
#   id                  CHAR(9) PK
#   partner_id          CHAR(9) FK -> referral_partners.id
#   simulation_id       CHAR(9) FK -> simulations.id
#   client_user_id      CHAR(9) FK -> users.id
#   simulation_charge   DECIMAL(8,2)  — amount the client was charged
#   commission_rate     DECIMAL(5,4)  — rate at time of charge (e.g. 0.2000)
#   commission_amount   DECIMAL(8,2)  — calculated commission
#   status              ENUM('pending','paid','refunded')
#   stripe_transfer_id  VARCHAR(255)  — populated on payout
#   created_at          DATETIME
#   paid_at             DATETIME NULL
# On Simulation refund: commission status -> 'refunded', not paid out.

15.9 Functional Requirements
FR-PAR-01: Admin can enable or disable the entire Referral Partner Program via the partner_program_enabled platform setting without redeployment.
FR-PAR-02: Admin can set and update the global commission rate (partner_commission_rate) and payout schedule (payout_schedule) at any time. Rate changes apply only to future commission events.
FR-PAR-03: Admin can approve, suspend, or deactivate any partner from the Admin Panel. Suspension immediately pauses commission accrual for that partner. Existing pending commissions are held, not forfeited.
FR-PAR-04: Each partner's referral link is unique, permanent, and tied to their referral_code. The code cannot be changed after approval.
FR-PAR-05: Attribution persists for 30 days from first referral link click via a server-side session cookie. If the user registers and generates a Simulation within this window, the referring partner is credited.
FR-PAR-06: Commission is calculated at the moment of Stripe payment confirmation, using the current simulation_price and partner_commission_rate at that instant. Both values are stored immutably on the commission record.
FR-PAR-07: If a Simulation is refunded, the associated commission record is marked 'refunded' and excluded from future payouts. If already paid out, the amount is deducted from the partner's next payout cycle.
FR-PAR-08: Payouts are only disbursed to partners with a completed Stripe Connect onboarding and an accumulated balance meeting or exceeding the min_payout_threshold.
FR-PAR-09: The Partner Dashboard client roster displays referred client full name, email address, signup date, number of Simulations generated, total commission earned from that client, and last activity date.
FR-PAR-10: Advisor view access is granted exclusively by the client. The advisor cannot request access — only receive it. The client can revoke access at any time, immediately and without notification delay.
FR-PAR-11: In advisor read-only view, the advisor sees all 5 layers fully expanded, all income streams with AI reasoning, and all agent action result panels and artifacts. No edit, regenerate, export, or reshare controls are visible.
FR-PAR-12: Coaching notes added by the advisor are visible to both the advisor and the client in the Simulation view. Notes are not visible to other collaborators or the platform admin.
FR-PAR-13: The client's Simulation view shows a 'Shared with Advisor' indicator when advisor access is active, displaying the advisor's name and the date access was granted.
FR-PAR-14: A referred client who is also a Referral Partner may refer other clients. There is no restriction on a user holding both a standard account and a partner account. Commission is earned on all downstream referrals regardless.
