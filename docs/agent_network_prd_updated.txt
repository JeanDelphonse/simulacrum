This PRD specifies a complete replacement of the four-step cycle diagram from SIM-PRD-VIZ-001 with a full seven-tier agent network visualization. The new diagram renders all nodes across the entire Layer 6 architecture simultaneously in a force-directed graph. Users can click any node to inspect its inputs and outputs for the selected cycle, and click any edge (connection arrow) to see the exact data payload that traveled between two agents. During active cycles, data flow is animated as traveling particles along edges in real time via SSE. Past cycles render as color-coded static snapshots.
1. Architecture — Seven Tiers, Full Node Catalog
1.1 Overview
The Agent Network Visualization renders all nodes and edges of the Simulacrum Layer 6 architecture in a single interactive canvas. The force-directed D3.js simulation positions nodes automatically based on connection strength — heavily connected nodes cluster toward the center, peripheral nodes settle at the edges. The user can drag any node to a preferred position; the layout remembers repositioned nodes for the session.
The diagram is not a schematic — it is a live operations view. Every node reflects the actual current status of that agent or component for the selected cycle. Every edge reflects whether data has flowed along that connection in the selected cycle and, when clicked, shows exactly what data flowed.
Caching principle applied to layout: the force simulation runs once on first render and its stable state is cached in a module-level variable. On subsequent cycle switches, only node status colors and edge weights are updated — the layout does not re-simulate. This preserves spatial memory: the user builds a mental map of the diagram layout and can rely on nodes being in the same position across cycle switches.

3. Force-Directed Layout Engine
3.1 D3 Force Configuration
The diagram uses D3.js v7 forceSimulation. The force configuration is tuned to reflect the architectural hierarchy: the ORC node is pulled to center, tier nodes cluster around it at appropriate distances, and peripheral nodes (connectors, audit log) settle at the outer ring. User drag repositions a node permanently for the session.
// d3_network.js — force simulation configuration
const TIER_Y = {
1: -280,  // L1-L5 agents — upper cluster
2:    0,  // ORC — center
3:  -80,  // Intelligence engines — inner ring
4:  120,  // GCC panels — lower right
5:  200,  // Integration connectors — outer ring
6: -200,  // Autonomy boundary — outer left
7:  240,  // Audit log — bottom
};
const simulation = d3.forceSimulation(nodes)
.force('link', d3.forceLink(edges)
.id(d => d.id)
.distance(d => edgeDistance(d))   // shorter = stronger pull
.strength(d => edgeStrength(d))
)
.force('charge', d3.forceManyBody()
.strength(d => d.id === 'ORC' ? -600 : -180)  // ORC repels more
)
.force('center', d3.forceCenter(svgWidth/2, svgHeight/2))
.force('tierY', d3.forceY(d => TIER_Y[d.tier] + svgHeight/2).strength(0.4))
.force('collision', d3.forceCollide().radius(d => nodeRadius(d) + 20))
.stop();
// Run headlessly to stable state (300 ticks)
for (let i = 0; i < 300; i++) simulation.tick();
// Cache stable positions
const nodePositions = {};
nodes.forEach(n => { nodePositions[n.id] = { x: n.x, y: n.y }; });
// Drag handler — updates cache on drag end
function dragEnded(event, d) {
nodePositions[d.id] = { x: event.x, y: event.y };
d3.select(this).classed('dragging', false);
}
function edgeDistance(d) {
// Shorter distance = stronger pull toward each other
if (d.source.id === 'ORC' || d.target.id === 'ORC') return 120;
if (d.source.tier === d.target.tier) return 80;
return 160;
}

4. Data Flow Inspector — Node & Edge Detail Panels
4.1 Inspector Panel Architecture
The inspector panel is a fixed-position slide-in drawer from the right side of the diagram container. It is 380px wide on desktop and full-screen on mobile. It renders different content depending on whether the user has clicked a node or an edge. The panel does not navigate away from the diagram — the diagram remains fully visible and interactive behind the panel.
The breadcrumb trail makes the inspector genuinely useful: when the user views an output value on a node panel, each output value has a small → icon. Clicking it closes the current panel, highlights the edge that carries that value, and opens the destination node's input panel showing where that value was consumed. This single interaction makes the entire data flow traceable without the user needing to memorize the topology.
4.2 Node Inspector Panel
The Node Inspector renders when the user clicks any node. It has four tabs: Overview, Inputs, Outputs, and History.

Clicking any row switches the diagram to that cycle's snapshot. The timeline makes it easy to identify cycles where a node behaved differently — e.g. a connector that was Stale for cycles 4 and 5 before recovering.

The Edge Inspector renders when the user clicks any edge. It shows the full data payload that traveled between the two connected nodes for the selected cycle.

Source node name → direction arrow → target node name. Step label (which of the four cycle steps this edge is active in: Harvest, Score, Schedule, or Report). Data flow status: Active (data flowed this cycle) / Idle (no data this cycle) / Error (flow failed).

Error detail
If the edge has Error status: the error message, timestamp, the connector's HTTP status code (for integration edges), and a 'Retry connection' button (for connector edges only — triggers a manual Harvest call for this connector).

5. Live Animation & SSE Data Flow
5.1 Active Cycle — Particle Animation
When the user has the diagram open during a live orchestrator cycle, data flow is animated as particles traveling along edge paths. Each particle represents a unit of data moving from source to target. The animation is driven by SSE events published by the Celery orchestration task.
// Particle animation — runs during live cycle
// Triggered by existing SSE events: 'dispatch' (ORC → layer agent) and
// 'action_complete' (layer agent → ORC). The edge to animate is derived
// from source_layer + action_type in the payload — no separate data_flow
// event is needed or published. Do NOT add a data_flow SSE event type.
function animateParticle(edgePath, edgeColor, valuePreview) {
const particle = svg.append('circle')
.attr('r', 5)
.attr('fill', edgeColor)
.style('filter', 'drop-shadow(0 0 4px ' + edgeColor + ')')
.style('opacity', 0.9);
// Travel along the SVG path
const pathLength = edgePath.getTotalLength();
particle.transition()
.duration(1800)           // 1.8s travel time
.ease(d3.easeLinear)
.attrTween('transform', () => t => {
const pt = edgePath.getPointAtLength(t * pathLength);
return `translate(${pt.x},${pt.y})`;
})
.on('end', () => {
// Flash the destination node on arrival
flashNode(edgePath.target.id);
// Update edge inspector payload if open
if (activeInspector?.edgeId === edgePath.id) {
refreshEdgeInspector(edgePath.id);
}
particle.remove();
});
}
// SSE events that trigger particle animation:
// 'harvest_pull'    — connector → ORC (one particle per connector)
// 'score_started'   — ORC → BAY, ORC → DAG, ORC → EXP
// 'score_complete'  — BAY → ORC, DAG → ORC, EXP → ORC
// 'dispatch'        — ORC → layer agent (per dispatched action)
// 'action_complete' — layer agent → ORC (per completed action)
// 'report_push'     — ORC → GCC (cycle summary)
// 'log_write'       — ORC → LOG (every event)
// NOTE: These map 1:1 to the SSE Event Specification table in Section 5.2.

10. Live Node Editor — Input Editor, Agent Re-run & Versioned Output
10.1 Overview
Clicking any agent action node in the swimlane diagram opens a slide-in editor panel from the right side of the diagram container. The panel shows the current input values for that agent action, pre-populated using the Pre-fill Engine (SIM-PRD-ART-001 Section 1). The user can edit any field and click Run to immediately re-execute the agent with the new values. The agent produces a new artifact version, the node receives a version badge in the diagram, and all downstream agents that depend on the updated artifact are marked stale and highlighted amber.
The diagram remains fully visible and interactive while the editor panel is open. The layout shifts to accommodate the 340px panel — the diagram container shrinks in width but does not navigate away. The panel closes via the × button, the Escape key, or by clicking a different node (which opens the editor for that node instead).
This feature connects three existing PRD systems: the Pre-fill Engine (SIM-PRD-ART-001 §1) provides the pre-populated field values and confidence scoring; Artifact Versioning (SIM-PRD-ART-001 §2) stores each re-run output as a new version; and Artifact-as-Input Dependency (SIM-PRD-ART-001 §4) drives the downstream stale propagation. The Node Editor is the user-facing control surface for all three systems working together.
10.2 Non-Sequential Routing — Architectural Principle
Agent output routing is non-sequential and data-driven. An agent action's artifact does not route exclusively to the next layer — it routes to any agent in any layer whose evaluation condition is satisfied by the output type and content. A Layer 1 Rate Card artifact can feed a Layer 3 Sales Page directly, or a Layer 5 Income Allocation agent, without passing through Layers 2 and 4. The routing decision is made by evaluating three properties of each completed artifact: output_type, evaluation condition, and timing. Artifact routing is non-exclusive — a single completed artifact may satisfy evaluation conditions for multiple downstream agents simultaneously. All satisfied downstream agents receive the routing signal in the same propagation pass, and the artifact is not consumed by the first recipient.
Timing type
Visual in diagram
Routing logic
Immediate
Solid teal line, arrowhead at target
Fires as soon as the agent completes and the evaluation condition is true. Example: Rate Card → Speaking Proposals fires immediately because output_type=pricing AND hourly_rate > 0.
Deferred
Dashed amber line, arrowhead at target
Fires when a specific threshold or condition is met after completion. Example: Waitlist Page → Launch Sequence fires only when page_live=true AND subscriber_count ≥ 50.
Periodic

# Body: { prefill_inputs: { field_key: value, ... } }
@bp.route('/api/simulations/<sim_id>/actions/<action_id>/rerun', methods=['POST'])
@login_required
def rerun_action(sim_id, action_id):
action     = Layer6ActionQueue.query.get_or_404(action_id)
new_inputs = request.get_json().get('prefill_inputs', {})
prior      = action.prefill_inputs or {}
# 1. Capture corrections → Bayesian feedback loop
for field, new_val in new_inputs.items():
if prior.get(field) != new_val:
db.session.add(PrefillCorrection(
id=generate_id(), simulation_id=sim_id,
user_id=current_user.id, action_type=action.action_type,
field_name=field, prefilled_value=str(prior.get(field)),
corrected_value=str(new_val), confidence_level='low'
))
# 2. Update action queue + dispatch
action.prefill_inputs = new_inputs
action.status = 'dispatched'
db.session.commit()
task = execute_agent_action.delay(sim_id, action_id, new_inputs)
# 3. SSE — diagram enters running state
redis_client.publish(f'layer6:{sim_id}', json.dumps({
'event_type': 'step_running',

}))

// diagram.js — SSE action_complete handler
sse.addEventListener('action_complete', e => {
const data = JSON.parse(e.data);
// 1. Version badge + artifact panel update

(data.stale_downstream || []).forEach(d => {

// 3. Animate edges carrying the updated artifact

});

A teal version badge (v2, v3...) appears on the node in the diagram immediately, sourced from the version_number field of the new artifact_versions record. The badge persists for the session.
FR-ANV-29: The artifact output section becomes visible after re-run, showing the artifact name, file type chips, and agent reasoning text. Reopening a previously run node pre-populates the artifact output from the stored artifact_versions record.
FR-ANV-30: After re-run, all downstream agents whose prefill_inputs depend on the updated artifact are marked stale: amber fill (#FFFAED), amber border (#E8B84B, 1.5px). The panel shows an alert listing each stale agent by layer, name, and evaluation condition. Stale clears when the user opens and re-runs that agent.
FR-ANV-31: Agent output routing is non-sequential and data-driven. Artifacts from any layer can route to any agent in any layer when the evaluation condition is satisfied. Skip routes (crossing ≥ 2 layer boundaries) render in purple dashed lines with an ↗ indicator. Filter controls allow isolation of skip, sequential, immediate, deferred, or periodic routes.
FR-ANV-32: Stale propagation is breadth-first and cascades up to 5 levels deep per FR-AD-06 from SIM-PRD-ART-001. Re-running a stale node propagates staleness to its own downstream agents in the same pass.
