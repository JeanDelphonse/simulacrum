Architecture — Seven Tiers, Full Node Catalog 1.1 Overview The Agent Network Visualization renders all nodes and edges of the Simulacrum Layer 6 architecture in a single interactive canvas. The force-directed D3.js simulation positions nodes automatically based on connection strength — heavily connected nodes cluster toward the center, peripheral nodes settle at the edges. The user can drag any node to a preferred position; the layout remembers repositioned nodes for the session. The diagram is not a schematic — it is a live operations view. Every node reflects the actual current status of that agent or component for the selected cycle. Every edge reflects whether data has flowed along that connection in the selected cycle and, when clicked, shows exactly what data flowed. Caching principle applied to layout: the force simulation runs once on first render and its stable state is cached in a module-level variable. On subsequent cycle switches, only node status colors and edge weights are updated — the layout does not re-simulate. This preserves spatial memory: the user builds a mental map of the diagram layout and can rely on nodes being in the same position across cycle switches. 1.2 The Seven Tiers — Node Catalog The diagram contains 26 named nodes organized across seven tiers. Each node has a type, a tier assignment, a color identity, and a defined set of inputs and outputs that the inspector renders when the node is clicked. Force-Directed Layout Engine 3.1 D3 Force Configuration The diagram uses D3.js v7 forceSimulation. The force configuration is tuned to reflect the architectural hierarchy: the ORC node is pulled to center, tier nodes cluster around it at appropriate distances, and peripheral nodes (connectors, audit log) settle at the outer ring. User drag repositions a node permanently for the session. // d3_network.js — force simulation configuration const TIER_Y = { 1: -280, // L1-L5 agents — upper cluster 2: 0, // ORC — center 3: -80, // Intelligence engines — inner ring 4: 120, // GCC panels — lower right 5: 200, // Integration connectors — outer ring 6: -200, // Autonomy boundary — outer left 7: 240, // Audit log — bottom }; const simulation = d3.forceSimulation(nodes) .force(&apos;link&apos;, d3.forceLink(edges) .id(d =&gt; d.id) .distance(d =&gt; edgeDistance(d)) // shorter = stronger pull .strength(d =&gt; edgeStrength(d)) ) .force(&apos;charge&apos;, d3.forceManyBody() .strength(d =&gt; d.id === &apos;ORC&apos; ? -600 : -180) // ORC repels more ) .force(&apos;center&apos;, d3.forceCenter(svgWidth/2, svgHeight/2)) .force(&apos;tierY&apos;, d3.forceY(d =&gt; TIER_Y[d.tier] + svgHeight/2).strength(0.4)) .force(&apos;collision&apos;, d3.forceCollide().radius(d =&gt; nodeRadius(d) + 20)) .stop(); // Run headlessly to stable state (300 ticks) for (let i = 0; i &lt; 300; i++) simulation.tick(); // Cache stable positions const nodePositions = {}; nodes.forEach(n =&gt; { nodePositions[n.id] = { x: n.x, y: n.y }; }); // Drag handler — updates cache on drag end function dragEnded(event, d) { nodePositions[d.id] = { x: event.x, y: event.y }; d3.select(this).classed(&apos;dragging&apos;, false); } function edgeDistance(d) { // Shorter distance = stronger pull toward each other if (d.source.id === &apos;ORC&apos; || d.target.id === &apos;ORC&apos;) return 120; if (d.source.tier === d.target.tier) return 80; return 160; } 3.2 Node Visual Specification Node State Visual Description Idle (no cycle selected) Circle, muted fill (30% opacity), grey label Default state when diagram first opens or when no cycle is selected. Data Flow Inspector — Node &amp; Edge Detail Panels 4.1 Inspector Panel Architecture The inspector panel is a fixed-position slide-in drawer from the right side of the diagram container. It is 380px wide on desktop and full-screen on mobile. It renders different content depending on whether the user has clicked a node or an edge. The panel does not navigate away from the diagram — the diagram remains fully visible and interactive behind the panel. The breadcrumb trail makes the inspector genuinely useful: when the user views an output value on a node panel, each output value has a small → icon. Clicking it closes the current panel, highlights the edge that carries that value, and opens the destination node&apos;s input panel showing where that value was consumed. This single interaction makes the entire data flow traceable without the user needing to memorize the topology. 4.2 Node Inspector Panel The Node Inspector renders when the user clicks any node. It has four tabs: Overview, Inputs, Outputs, and History. Tab Contents Overview Node name and tier. Current status badge (Idle / Running / Complete / Error / Escalated). For layer agents: count of actions dispatched this cycle, count completed, count failed. For ORC: cycle number, phase (Explore/Exploit), actions scored, dispatched, escalated. For engine nodes: last computation timestamp and a one-line result summary. For connectors: last pulled timestamp, connection status (Live / Stale / Failed), and the name of the integration account connected. Inputs A structured key-value list of every field this node received as input for the selected cycle. Field name on the left in bold navy. Value on the right in dark text. Timestamps shown in relative format (&apos;14 minutes ago&apos;). If the input came from another node, a small &apos;From [Node ID]&apos; chip is shown — clicking it opens that node&apos;s Output tab. Scrollable if input count is large. A &apos;Copy all as JSON&apos; button at the bottom exports the full input payload. Outputs A structured key-value list of every field this node produced as output in the selected cycle. Same format as Inputs. Each output value has a small → icon — clicking it highlights the edge that carries that value outbound and opens the receiving node&apos;s Input tab, showing where the value was consumed. A &apos;Copy all as JSON&apos; button at the bottom. History A timeline of this node&apos;s status across the last 10 cycles. Each cycle shown as a row: cycle number, date, status, and a mini summary (e.g. &apos;L1: 3 actions dispatched, 3 complete&apos;). Clicking any row switches the diagram to that cycle&apos;s snapshot. The timeline makes it easy to identify cycles where a node behaved differently — e.g. a connector that was Stale for cycles 4 and 5 before recovering. 4.3 Edge Inspector Panel The Edge Inspector renders when the user clicks any edge. It shows the full data payload that traveled between the two connected nodes for the selected cycle. Section Contents Edge header Source node name → direction arrow → target node name. Step label (which of the four cycle steps this edge is active in: Harvest, Score, Schedule, or Report). Data flow status: Active (data flowed this cycle) / Idle (no data this cycle) / Error (flow failed). Payload A structured key-value table of every field in the edge&apos;s data payload for the selected cycle. Field name (left column, bold), value (middle column), type (right column — string, int, float, datetime, array, JSON). Array and JSON values are collapsible — collapsed by default showing item count, expand to see full content. A &apos;Copy payload as JSON&apos; button at the top right. Value trace Each value in the payload has a dual trace button: an ← button (&apos;Where did this come from?&apos;) that navigates to the source node&apos;s Output tab showing how this value was produced, and a → button (&apos;Where does this go?&apos;) that navigates to the target node&apos;s Input tab showing how this value is consumed. This makes the full data lineage traceable in a click chain. Payload history A mini timeline below the payload: this edge&apos;s last 5 cycle payloads shown as collapsed rows (cycle number + one key metric). The user can expand any prior cycle to see its full payload. Useful for spotting when a connector value changed significantly between cycles (e.g. Apollo reply rate dropped from 28% in cycle 3 to 9% in cycle 4). Error detail If the edge has Error status: the error message, timestamp, the connector&apos;s HTTP status code (for integration edges), and a &apos;Retry connection&apos; button (for connector edges only — triggers a manual Harvest call for this connector). 4.4 Breadcrumb Trail — Value Lineage Navigation The breadcrumb trail is the feature that makes the inspector genuinely powerful. Every value in the Node Inspector or Edge Inspector is traceable from its origin through every node it passes through to its final consumption point. The user builds this trail by clicking the ← and → trace buttons. // Example breadcrumb navigation sequence: // // User clicks APO node → Opens Node Inspector → Outputs tab // Sees: open_rate: 0.32 // Clicks → on open_rate // → Highlights APO→ORC edge // → Opens Edge Inspector for APO→ORC // → Shows payload including open_rate: 0.32 // Clicks → (Where does this go?) // → Opens ORC Node Inspector → Inputs tab // → Shows open_rate: 0.32 received from APO // Clicks → on open_rate // → Highlights ORC→BAY edge // → Opens Edge Inspector for ORC→BAY // → Shows layer6_momentum snapshot including open_rate: 0.32 // Clicks → (Where does this go?) // → Opens BAY Node Inspector → Inputs tab // → Shows open_rate: 0.32 as one input to the Bayesian model // Clicks → on open_rate // → Opens BAY → Outputs tab // → Shows posterior_mean for L1 Active Income stream // → User sees how the 0.32 open rate influenced the L1 score // // Full lineage: APO.output → APO→ORC edge → ORC.input // → ORC→BAY edge → BAY.input → BAY.output (L1 score) // Breadcrumb bar: APO &gt; APO→ORC &gt; ORC &gt; ORC→BAY &gt; BAY // Breadcrumb bar renders at top of inspector panel: // [APO] &gt; [APO→ORC] &gt; [ORC] &gt; [ORC→BAY] &gt; [BAY] // Each item is clickable — navigates back to that point in the trail. // Maximum breadcrumb depth: 8 hops before trail resets. 5. Each particle represents a unit of data moving from source to target. The animation is driven by SSE events published by the Celery orchestration task. // Particle animation — runs during live cycle // Triggered by SSE event: &apos;data_flow&apos; with payload {edge_id, value_preview} function animateParticle(edgePath, edgeColor, valuePreview) { const particle = svg.append(&apos;circle&apos;) .attr(&apos;r&apos;, 5) .attr(&apos;fill&apos;, edgeColor) .style(&apos;filter&apos;, &apos;drop-shadow(0 0 4px &apos; + edgeColor + &apos;)&apos;) .style(&apos;opacity&apos;, 0.9); // Travel along the SVG path const pathLength = edgePath.getTotalLength(); particle.transition() .duration(1800) // 1.8s travel time .ease(d3.easeLinear) .attrTween(&apos;transform&apos;, () =&gt; t =&gt; { const pt = edgePath.getPointAtLength(t * pathLength); return `translate(${pt.x},${pt.y})`; }) .on(&apos;end&apos;, () =&gt; { // Flash the destination node on arrival flashNode(edgePath.target.id); // Update edge inspector payload if open if (activeInspector?.edgeId === edgePath.id) { refreshEdgeInspector(edgePath.id); } particle.remove(); }); } // SSE events that trigger particle animation: // &apos;harvest_pull&apos; — connector → ORC (one particle per connector) // &apos;score_send&apos; — ORC → BAY, ORC → DAG, ORC → EXP // &apos;score_result&apos; — BAY → ORC, DAG → ORC, EXP → ORC // &apos;dispatch&apos; — ORC → layer agent (per dispatched action) // &apos;action_result&apos; — layer agent → ORC (per completed action) // &apos;report_push&apos; — ORC → GCC (cycle summary) // &apos;log_write&apos; — ORC → LOG (every event) 5.2 SSE Event Specification SSE Event Type Payload Diagram Response cycle_started cycle_number, phase, timestamp ORC node → Running state (pulse ring).
