PRODUCT REQUIREMENTS DOCUMENT
Unified Integration & Signal Specification
9 Platforms · Bayesian Signal Taxonomy · Feedback Loop Architecture · Revenue Attribution
Document ID
SIM-PRD-INTEG-001
Version
1.0 — Comprehensive integration specification
Scope
Nine integration platforms, each with: connection architecture, deployment functions, webhook/signal capture, Bayesian signal taxonomy with weights and update formulas, and functional requirements. Plus a unified cross-integration signal taxonomy in Section 10.
The nine platforms
1. Apollo (email outreach) · 2. Stripe Connect (payments) · 3. Cal.com (bookings) · 4. PandaDoc (e-signatures) · 5. Kajabi (courses/memberships) · 6. ConvertKit (email lists) · 7. LinkedIn (profile/content/connections) · 8. Plaid (financial accounts) · 9. Alpaca (brokerage/trade execution)
Core principle
Every integration has a feedback loop. Signals flow from webhooks into the Bayesian orchestrator's posterior updates. The orchestrator uses these signals to decide which agents to dispatch in the next cycle. No integration is connect-and-forget — every one contributes data that makes the system smarter.
Revenue chain
Apollo sends emails → replies advance contacts → Cal.com books calls → PandaDoc sends proposals → Stripe collects payment → income captured → Bayesian model updated → next cycle dispatches more of what worked. Every integration is a link in this chain.
Companion doc
This document specifies per-integration signals. SIM-PRD-SIGNAL-001 (Section 10 of this document) is the unified cross-integration signal taxonomy that the orchestrator references for all Bayesian updates.

This is the largest single PRD in the Simulacrum suite. It specifies every external integration the platform touches, with a consistent structure per integration: how the user connects it, how agents deploy through it, what signals it sends back, and exactly how those signals update the Bayesian model. Section 10 brings all signals together into a unified taxonomy — the orchestrator's single source of truth for every posterior update across every integration.
1. Apollo — Email Outreach
Purpose: Send cold email campaigns, consulting outreach, speaking pitches, corporate training pitches, alumni reactivation, and referral network introductions via Apollo sequences.
Primary income layer: Layer 1 (Active income) and Layer 2 (Leveraged income)
1.1 Connection Architecture
User connects their own Apollo account via OAuth 2.0 from Settings → Integrations. Simulacrum receives a scoped access token to create contacts, sequences, and webhooks on the user's Apollo account. Emails are sent from the user's email address via Apollo — not from a Simulacrum-managed account.
# Apollo OAuth scopes requested:
scopes: sequences:write, contacts:write, emailer_campaigns:write, webhooks:write
# Stored in user_integrations:
integration_type = 'apollo'
auth_type        = 'oauth'
access_token     = encrypt(apollo_access_token)
refresh_token    = encrypt(apollo_refresh_token)
account_id       = apollo_account_id
status           = 'connected' | 'expired' | 'disconnected'

1.5 Functional Requirements — Apollo
FR-APOLLO-01: User connects Apollo via OAuth 2.0. Access token encrypted in user_integrations. Token refresh handled automatically. Expired tokens trigger escalation action item (tier 1) in the GCC.
FR-APOLLO-02: Six agent action_types deploy through Apollo per Section 1.2. cold_email_campaign creates a 3-step sequence. consulting_outreach, speaking_proposals, and corporate_training_pitch send individual emails. alumni_reactivation creates segmented sequences. referral_network creates an intro sequence.
FR-APOLLO-03: Apollo webhooks are registered per sequence/send on deployment. Six event types are handled: email_sent, email_opened, email_replied, email_bounced, email_unsubscribed, email_clicked. Each event creates a contact_activities record and updates the relevant email_campaigns counter.
FR-APOLLO-04: Ten Bayesian signals are captured from Apollo events per Section 1.4. Each signal has an explicit weight, direction (+/-), posterior target, and EMA update formula. The orchestrator's Score step reads these posteriors to weight future Apollo-dependent action dispatches.
FR-APOLLO-05: Per-step attribution: when a reply arrives, the system records which sequence step triggered it (step 1, 2, or 3). This feeds the step_effectiveness posterior, which informs future sequence generation (e.g., if step 3 replies dominate, the agent generates more aggressive follow-up copy).
FR-APOLLO-06: Compound signals: reply → booking and reply → client are tracked as cross-integration signals linking Apollo reply events to Cal.com booking events and Stripe payment events via the contact_id join. These carry the highest weights (0.7 and 1.0 respectively).
FR-APOLLO-07: Daily send limit is configurable per user in Settings → Integrations (default 30). Apollo sequence daily_limit parameter is set at sequence creation. New Apollo account warmup: limit starts at 10/day for the first week, 20/day for week 2, full limit from week 3. Warmup cadence is enforced by Simulacrum, not by Apollo.
FR-APOLLO-08: CAN-SPAM compliance: physical business address appended to every email footer. One-click unsubscribe link managed by Apollo. Subject line deceptive pattern check before deployment. GDPR legitimate interest basis for EU contacts (attorney review required).

2.5 Functional Requirements — Stripe
FR-STRIPE-01: User connects Stripe via Connect Standard OAuth. Access token and connected account ID encrypted in user_integrations. Disconnection does not affect existing payment links — they remain active in the user's Stripe account.
FR-STRIPE-02: Seven agent action_types create Stripe objects per Section 2.2. Every Stripe object includes the full metadata set: simulacrum_simulation_id, layer_number, action_type, action_id, artifact_id, income_type, user_id.
FR-STRIPE-03: Seven webhook events are handled per Section 2.3. Stripe signature verification is required on every webhook. Metadata is read from the payment object to attribute income.
FR-STRIPE-04: Ten Bayesian signals are captured per Section 2.4. payment_intent.succeeded and invoice.paid carry weight 1.0 — the highest in the system. Refunds carry weight 0.8 negative. Disputes carry weight 0.9 negative.
FR-STRIPE-05: Revenue velocity is a computed compound signal: days from first CRM contact creation to first Stripe payment for that contact's email. Shorter velocity increases the action_type's ROI posterior.
FR-STRIPE-06: Subscription churn (subscription.deleted) feeds the churn_rate posterior per action_type. An action_type with churn_rate above 15% in the trailing 90 days gets its dispatch probability reduced by 30%.
FR-STRIPE-07: Average transaction value is tracked per layer as a running EMA(alpha=0.2). Used by the rate_card agent as a pricing anchor input — if actual transaction values consistently exceed the rate card's suggested prices, the agent recommends a rate increase.

3.5 Functional Requirements — Cal.com
FR-CAL-01: User connects Cal.com via OAuth 2.0. Three agent action_types deploy through Cal.com per Section 3.2. Webhooks registered per event type for four event types: BOOKING_CREATED, BOOKING_CANCELLED, BOOKING_RESCHEDULED, MEETING_ENDED.
FR-CAL-02: BOOKING_CREATED advances the attendee's CRM contact from pipeline_stage='prospect' to 'active'. Creates contact_activities record. Increments layer6_momentum.consulting_bookings_mo. Publishes harvest_complete SSE to GCC.
FR-CAL-03: BOOKING_CANCELLED creates a contact_activities record with activity_type='booking_cancelled'. Decrements the momentum metric. Updates cancellation_rate posterior with weight 0.4 negative.
FR-CAL-04: No-show detection: if no MEETING_ENDED event arrives within 24 hours of a BOOKING_CREATED event's scheduled time, the system flags the booking as a no-show. Creates contact_activities record with activity_type='no_show'. Updates noshow_rate posterior with weight 0.5 negative.
FR-CAL-05: Nine Bayesian signals captured per Section 3.4 including three compound signals (booking→proposal, booking→client, time_of_day_performance) that require cross-integration joins with PandaDoc and Stripe via contact_id.
FR-CAL-06: Session type performance comparison: each Cal.com event type tracks its own booking_rate, cancellation_rate, noshow_rate, and booking_to_client_rate independently. The booking_page agent uses these to recommend which session types to keep, modify, or remove.

10. Unified Cross-Integration Signal Taxonomy
10.1 How the Orchestrator Reads Signals
The orchestrator's Score step reads all active Bayesian posteriors from all integrations. Each posterior is a probability estimate for how well a specific action_type converts for this user. The posteriors are updated via the Exponential Moving Average (EMA) formula whenever a signal arrives from any integration.
# Bayesian posterior update — Exponential Moving Average
# Applied to every signal across all integrations
def update_posterior(posterior_key, signal_value, weight, direction):
    """
    posterior_key: e.g. 'reply_rate:cold_email_campaign'
    signal_value: the observed rate (e.g. 0.16 for 16% reply rate)
    weight: per Section X.4 of each integration (e.g. 0.5 for email_replied)
    direction: '+' or '-'
    """
    current = get_posterior(posterior_key)  # from layer6_outcomes
    alpha = weight * 0.3   # EMA alpha scaled by signal weight
    # Higher-weight signals shift the posterior more aggressively
    if direction == '+':
        new_value = alpha * signal_value + (1 - alpha) * current
    else:  # negative signal
        new_value = alpha * (1 - signal_value) + (1 - alpha) * current
    set_posterior(posterior_key, new_value)
    log_signal(posterior_key, signal_value, weight, direction)

10.4 The Revenue Chain
The compound signals form a chain. When the orchestrator sees the full chain fire for a specific user — Apollo reply → Cal.com booking → PandaDoc signed → Stripe payment confirmed — it has maximum confidence in the L1 consulting pipeline and will aggressively dispatch more consulting outreach in the next cycle. When any link in the chain breaks (high reply rate but low booking rate), the orchestrator identifies the weak link and dispatches agents to fix it (regenerate the booking page with different session types, update the discovery call pre-questionnaire).
This is what makes Simulacrum fundamentally different from a content generation tool. The system does not just generate artifacts — it observes which artifacts produce revenue and concentrates effort on the proven paths. Every integration is a sensor. Every webhook is a data point. The Bayesian model is the brain that turns data points into dispatch decisions. Without the feedback loop, the system is a template generator. With it, the system learns.
