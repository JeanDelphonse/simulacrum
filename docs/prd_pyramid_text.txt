This page serves two purposes in one view. First, it shows the user what the AI found — income opportunities grounded in their resume, with revenue estimates, channels, and reasoning. Second, it lets the user act on those opportunities immediately — selecting agents, configuring their parameters, and running them. The user never has to navigate away from the simulation page to start executing. Strategy and execution live on the same screen.
1. Page Architecture

4.6 Run Agent Flow
When the user clicks 'Run Agent' (or 'Re-run Agent'):
The panel closes immediately. The agent card in the grid transitions to 'RUNNING' status (amber badge, Clock icon, 'Running...' text).
A POST request is sent to /api/simulations/:id/agents/:action_type/run with the configured parameters as the request body.
The server creates an ActionQueueItem with status='pending', stores the parameters as prefill_inputs on the action record, and dispatches the agent as a Celery task (same execution path as orchestrator dispatch per SIM-PRD-ORCH-001 v2.0 Section 5).
The DAG is validated server-side before dispatch. If prerequisites are not met, the request returns a 422 error with the missing prerequisites listed. The panel re-opens with an error banner: 'Cannot run — [prerequisite agent] must complete first.'
If the agent requires an integration that is not connected, the request returns a 422 with the missing integration. The panel re-opens with an error banner: 'Connect [integration] in Settings to run this agent.' Link to Settings → Integrations.
On successful dispatch, a toast notification appears: '[Agent name] is running — you will see the result in your Action Queue.' The user can continue browsing the page.
When the agent completes (Celery task finishes), the card updates to 'DONE' status via a WebSocket push or polling. The 'View artifact →' link becomes active.

5.2 Agent Status Query
# For each layer, query the current status of all agents
def get_agent_statuses(simulation_id, layer_number):
agents = LAYER_AGENTS[layer_number]  # all agents for this layer
statuses = {}
for agent_type in agents:
latest = ActionQueueItem.query.filter_by(
simulation_id=simulation_id,
action_type=agent_type,
).order_by(ActionQueueItem.created_at.desc()).first()
if latest and latest.status == 'complete':
statuses[agent_type] = 'complete'
elif latest and latest.status == 'in_progress':
statuses[agent_type] = 'in_progress'
elif check_prerequisites_exist(agent_type, simulation_id):
statuses[agent_type] = 'available'
else:
statuses[agent_type] = 'locked'
return statuses
5.3 Run Agent API
@bp.route('/api/simulations/<sim_id>/agents/<action_type>/run', methods=['POST'])
@login_required
def run_agent(sim_id, action_type):
sim = Simulation.get_or_404(sim_id)
if sim.user_id != current_user.id:
abort(403)
# Validate DAG prerequisites
prereqs = AGENT_DAG.get(action_type, [])
missing = [p for p in prereqs if not artifact_exists(sim_id, p)]
if missing:
return jsonify({'error': 'prerequisite_missing',
'missing': missing}), 422
# Validate integration availability
integrations = AGENT_INTEGRATION_MAP.get(action_type, [])
for integ in integrations:
if integ != 'crm' and not has_integration(sim_id, integ):
return jsonify({'error': 'integration_missing',
'integration': integ}), 422
# Create action record with user-provided parameters
params = request.json.get('params', {})
action = ActionQueueItem.create(
simulation_id=sim_id,
action_type=action_type,
status='pending',
prefill_inputs=json.dumps(params),
source='manual',  # distinguishes from orchestrator-dispatched
)
# Dispatch Celery task
task = execute_agent.delay(sim_id, action_type, cycle_id=None)
action.celery_task_id = task.id
db.session.commit()
return jsonify({'status': 'dispatched', 'action_id': action.id}), 200
6. Functional Requirements
FR-PYR-01: The Wealth Pyramid Launchpad renders at /simulations/:id. It is the simulation detail page — the user lands here after creating a simulation or selecting one from the Dashboard.
FR-PYR-02: The page header shows simulation status (Complete/Generating/Paused), creation date, page title ('Your Wealth Pyramid'), and three action buttons: Export PDF, Share, and Growth Command Center (links to /simulations/:id/gcc).
FR-PYR-03: Five layers render as a single-open accordion. Default: L1 expanded. Each layer header shows: layer icon, badge (L1-L5), title + subtitle, agent status counts (done/running/ready), priority badge (HIGH/MEDIUM), and 4px left border in layer color.
FR-PYR-04: Part 1 (AI Analysis) occupies the top of each expanded layer. It contains: the positioning paragraph in a teal-bordered box, 3-6 opportunity cards with service name, revenue range, description, channel tags, and expandable AI reasoning with evidence bullets and automation/launch tags. A 'Refine this layer' button opens the Simi co-pilot with a pre-filled prompt.
FR-PYR-05: Part 2 (Agent Grid) occupies the bottom of each expanded layer, separated by a 2px horizontal rule. It contains: section header ('Execute — All L[N] Agents' with agent counts) and a responsive grid of ALL agents for that layer. No agents are hidden — locked agents are dimmed but visible.
FR-PYR-06: Agent cards show: icon, status badge (Done/Running/Ready/Locked), agent name, one-line description, and a status-dependent action link (Configure & Run / View artifact / Running / Prerequisites needed). Available cards have a teal-highlighted border.
FR-PYR-07: Clicking an available or completed agent card opens a configuration slide-over panel (440px wide, slides from right). The panel shows: agent name, description, completion banner (if applicable), configurable parameters (select, text, toggle, textarea), integration info ('Deploys to' pills), and Run/Re-run button.
FR-PYR-08: Agent parameters are defined per agent per Section 4.4. Parameters are stored as prefill_inputs on the ActionQueueItem record. Agents with no parameters show a placeholder card and run with simulation defaults.
FR-PYR-09: Run Agent dispatches a POST to /api/simulations/:id/agents/:action_type/run. The server validates DAG prerequisites and integration availability before dispatching. Missing prerequisites or disconnected integrations return 422 with actionable error messages. Successful dispatch returns 200 and creates a Celery task.
FR-PYR-10: Manual agent runs (source='manual') follow the same execution path as orchestrator-dispatched runs: same Celery task, same output validation, same integration deployment, same bio page auto-update, same CRM saves. The only difference is source attribution — the Bayesian model tracks manual vs orchestrator dispatch separately.
FR-PYR-11: All 49 agents are represented across the 5 layer grids: L1 (11), L2 (8), L3 (10), L4 (10), L5 (10). The grid shows every agent the platform supports. Locked agents display 'Prerequisites needed' so the user understands what unlocks them.
FR-PYR-12: Re-running a completed agent creates a new artifact version (per SIM-PRD-VIEW-001 version history). The previous version is retained. The user can view and compare versions in the artifact Full View.
FR-PYR-13: When an agent completes, its card updates from 'Running' to 'Done' via WebSocket push (preferred) or polling (fallback, every 10 seconds while any agent is running). A toast notification appears: '[Agent name] completed — view artifact.'
